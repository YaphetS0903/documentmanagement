# 文档管理平台正式上线与数据迁移实施方案

## 1. 目标和原则

本方案用于把旧开始云/KASS 文档资料迁入新 B/S 文档管理平台，并完成生产环境上线。迁移遵循以下原则：

- 旧文件源只读，迁移工具不修改或删除源目录内容。
- 先扫描、再演练、后执行；没有 `--apply` 时绝不写入平台。
- 不直接修改 MySQL 快照表，所有目录和文件通过平台 API 写入，确保权限、版本、审计和索引流程完整执行。
- 迁移凭据只从环境变量读取，不写入报告、日志、命令参数或断点文件。
- 每批迁移前创建平台备份，迁移后进行数量、容量、抽样内容和权限验收。

## 2. 上线阶段

### 阶段 A：业务确认

确认并形成签字清单：

1. 迁移全部文件还是仅迁移有效文件。
2. 是否保留旧目录原样，或映射到“01 董事会、02 TES、03 公司管理、04 知识管理、05 公司大事记”。
3. 同名文件冲突采用跳过还是上传为新版本。
4. 哪些旧账号继续使用，哪些账号停用。
5. TES 四级文件及其他目录的预览、打印、下载、修改和外链权限。
6. 历史版本、评论和旧操作日志是否有可导出的结构化数据；仅有文件目录时只能迁移当前文件正文。

### 阶段 B：生产环境准备

- Node.js 20 LTS、Nginx、HTTPS 和进程守护配置完成。
- MySQL 正式库连接成功，系统运行状态不得显示 JSON 降级。
- `DATA_DIR`、`UPLOAD_DIR`、`TMP_DIR`、`BACKUP_DIR`、`QUARANTINE_DIR` 位于持久化磁盘。
- ONLYOFFICE、JWT、企业微信、邮件或其他身份源按实际范围配置。
- 生产管理员使用强密码，演示账号停用。
- 执行 `npm run production:check` 和 `npm run release:check`。

### 阶段 C：试迁移

先选一个不超过正式数据 5% 的代表性目录，包含 Word、Excel、PPT、PDF、图片、代码、中文长文件名和多级目录。完成扫描、演练、执行和验收后，再开始全量迁移。

### 阶段 D：全量迁移和增量补迁

1. 旧系统进入只读窗口。
2. 创建完整平台备份并完成恢复演练。
3. 执行全量迁移，保存报告和断点文件。
4. 对试迁移后新增或修改的旧文件执行增量补迁。
5. 完成业务抽样验收后切换用户入口。

## 3. 迁移工具

### 3.1 扫描预检

```bash
npm run migration:scan -- \
  --source /data/old-document-library \
  --output /data/migration/scan-report.json \
  --exclude '*.tmp' \
  --exclude '回收站/**' \
  --hash
```

工具同时生成 JSON 和 Markdown 报告，内容包括文件数、目录数、总容量、扩展名分布、不可读文件、非法名称、大小写冲突、长路径和符号链接。

必须先处理所有 `error` 级问题。`warning` 需人工确认，但不会默认阻止迁移。

### 3.2 创建迁移凭据

建议在“开放 API”中创建临时迁移凭据，迁移结束后立即禁用。凭据通过环境变量提供：

```bash
export MIGRATION_ACCESS_KEY='临时 Access Key'
export MIGRATION_ACCESS_SECRET='临时 Secret'
```

也可短期使用：

```bash
export MIGRATION_TOKEN='管理员短期 JWT'
```

不要把凭据写进 shell 脚本、命令历史文件、迁移报告或 Git 仓库。

### 3.3 演练

```bash
npm run migration:import -- \
  --source /data/old-document-library \
  --base-url https://docs.example.com \
  --target-parent n_root \
  --state /data/migration/migration-state.json \
  --conflict skip
```

没有 `--apply` 时只检查平台连接、上传策略和迁移规模，不创建目录或上传文件。

### 3.4 正式执行

```bash
npm run migration:import -- \
  --source /data/old-document-library \
  --base-url https://docs.example.com \
  --target-parent n_root \
  --root-folder '旧系统归档' \
  --state /data/migration/migration-state.json \
  --conflict skip \
  --continue-on-error \
  --apply
```

说明：

- 默认在目标目录下创建与迁移源同名的根文件夹；可用 `--root-folder` 指定名称，或 `--no-root-folder` 直接写入目标目录。
- `--conflict skip`：目标存在同名文件时跳过，适合首次全量和重复执行。
- `--conflict version`：目标存在同名文件时上传为新版本；超过 250MB 的冲突文件需人工处理。
- 大文件自动使用分片上传，小文件使用普通上传。
- 每成功一个文件就原子更新断点文件。命令中断后使用相同源、目标和断点文件重新运行即可续跑。
- `--continue-on-error` 会记录失败并继续后续文件；不加时遇到首个失败即停止。

## 4. 目录和权限初始化

建议先在平台创建业务一级目录，再按目录分别执行迁移：

| 旧资料范围 | 新平台目标目录 | 默认权限建议 |
|---|---|---|
| 董事会资料 | 01 董事会 | 指定董事会和管理层可见 |
| TES 体系文件 | 02 天瑞卓越体系（TES） | 全员预览；四级表单按业务授予打印 |
| 部门管理文件 | 03 公司管理 | 部门继承权限，文档管理员维护 |
| 培训和知识资料 | 04 知识管理 | 全员预览，指定人员维护 |
| 公司事件资料 | 05 公司大事记 | 全员预览，管理员维护 |

迁移文件前先配置目录权限；不要先给全员完全控制，再依赖迁移后逐个收紧。

## 5. 验收标准

### 数据完整性

- 扫描报告文件数与平台迁移成功、跳过、失败数量可对账。
- 平台当前版本总容量与源目录容量差异有明确解释。
- 随机抽查至少 50 个文件，覆盖主要扩展名和各一级目录。
- 对启用 `--hash` 的批次，抽样比对 MD5。

### 功能验收

- DOCX/XLSX/PPTX 原版预览和在线编辑正常。
- PDF、图片、文本和代码预览正常。
- 搜索能命中文件名和可索引正文。
- 普通员工看不到未授权目录，管理员权限预览结果正确。
- 下载、打印、外链和敏感文档水印符合目录策略。
- 手机端和企业微信完成真实账号验收。

### 运维验收

- MySQL、ONLYOFFICE、磁盘和同步目录健康检查正常。
- 完整备份可下载，恢复演练通过。
- 迁移期间没有持续增长的失败告警或隔离文件。
- 日志中可查询迁移账号产生的目录创建和文件上传记录。

## 6. 回滚方案

迁移只新增平台数据，不修改源目录。出现阻断问题时：

1. 停止迁移命令并保留断点文件和日志。
2. 未切换用户入口时，直接修复问题后续跑。
3. 已切换但数据不可用时，恢复发布前平台备份和应用版本。
4. 恢复完成后执行一致性检查、搜索索引重建和核心流程冒烟测试。
5. 原因未确认前不要删除旧系统或旧文件源。

## 7. 上线交付物

- 业务确认后的目录映射和权限矩阵。
- 扫描 JSON/Markdown 报告。
- 迁移断点状态文件和最终执行汇总。
- 发布前完整备份及恢复演练记录。
- 管理员、普通员工、企业微信和移动端验收记录。
- 正式版本号、Git 提交、标签和回滚版本记录。
